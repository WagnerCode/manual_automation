# Документация по автоматизации развёртывания кластера Corax в Cloud.Ru

Данный проект предназначен для автоматизированного создания инфраструктуры и развёртывания кластера Corax (на базе Kafka и Zookeeper) в облаке Cloud.Ru.

---

## 1. 📘 Обзор проекта

### Назначение
Автоматизация решает задачи:
*   Подготовки сетевой и вычислительной инфраструктуры в Cloud.Ru.
*   Конфигурации операционной системы (Alt Linux) и подготовки дисковых подсистем (LVM).
*   Ввода узлов в домен FreeIPA и выпуска SSL-сертификатов.
*   Установки и настройки компонентов Corax.

### Архитектура и взаимодействие
Решение построено на конвейерной обработке данных:
1.  **Python (Подготовка)**: Скрипт `ansible_prepare_021.py` читает исходные данные из каталога `INV/`, рассчитывает сетевые параметры и генерирует YAML-инвентарь для Ansible.
2.  **Ansible (Оркестрация)**: Основной плейбук `playbook.yaml` управляет этапами развёртывания.
3.  **Python (Валидация)**: Скрипт `check_entity.py` вызывается из Ansible для проверки наличия ресурсов (VM, Subnets, Flavors) через Cloud.Ru API.
4.  **Terraform (Инфраструктура)**: Ansible генерирует `.tf` файлы на основе шаблонов и вызывает Terraform для создания ресурсов.
5.  **Ansible (Конфигурация)**: Настройка ОС и установка Corax.

### Data Flow
`INV/*.csv` → `ansible_prepare_021.py` → `Ansible Inventory` → `Ansible Roles` → `Terraform Templates` → `Cloud.Ru API`

## 2. 🛠 Предварительные требования

### Программное обеспечение
*   **Python 3.x**: с библиотеками `requests`, `pyyaml`.
*   **Ansible**: версия 2.10+.
*   **Terraform**: версия 1.0+ с провайдером `cloud.ru/cloudru/cloud` v2.0.0.
*   **Утилиты**: `sshpass`, `jq`, `openssl`, `keytool` (Java 17).

### Переменные окружения (обязательные)
| Переменная | Описание |
| :--- | :--- |
| `TF_VAR_CLOUDRU_KEY_ID` | Идентификатор ключа доступа Cloud.Ru |
| `TF_VAR_CLOUDRU_SECRET` | Секретный ключ доступа Cloud.Ru |
| `CLOUDRU_PROJECT_ID` | ID проекта в Cloud.Ru |
| `IPA_ADDER` | Имя пользователя для ввода в домен IPA |
| `IPA_ADDER_PASSWORD` | Пароль пользователя IPA |
| `CLOUDRU_*_API_URL` | URL-адреса различных API сервисов (Compute, VPC, IAM и т.д.) |

### Требования к доступам
*   **Сетевой доступ**: Доступ к API Cloud.Ru (`*.api.cloud.ru`).
*   **SSH**: Возможность подключения к создаваемым ВМ. В коде зашит `ProxyJump=d.kulida@10.246.8.22`.
*   **IPA**: Доступ к серверам FreeIPA для регистрации хостов.

## 3. 📁 Структура инвентаря и подготовка данных

Данные хранятся в каталоге `INV/`. Структура каталогов определяет иерархию групп Ansible.

### Основные файлы в INV/:
*   **`vm_params.csv`**: Параметры ВМ (CPU, RAM, диски, образ, AZ, проект).
*   **`net_params.csv`**: Сетевые параметры кластера (подсеть, GIS подсеть).
*   **`ipa_params.csv`**: Параметры домена IPA.
*   **`dk_ssh_key.csv`**: SSH-ключи для добавления в `authorized_keys`.
*   **Подкаталоги (напр. `corax_aaaah/`)**: Содержат описания конкретных хостов.
    *   `ip`: Файл с IP-адресом управления.
    *   `access.txt`: Данные доступа (user, password, become_pass, port).
    *   `eth0.csv`: Параметры сетевого интерфейса.

### Логика работы `ansible_prepare_021.py`:
1.  **Рекурсивный обход**: Скрипт проходит по всем папкам. Если находит файл `ip`, считает папку хостом.
2.  **Расчёт параметров**:
    *   Автоматически вычисляет Gateway, если в CSV указан порядковый номер (напр. `gateway,1` -> первый IP подсети).
    *   Определяет `prefixlen` на основе маски.
3.  **Генерация маршрутов**: На основе параметра `route_target` формирует содержимое файлов `route-ethX`.
4.  **Сбор hosts**: Формирует список для `/etc/hosts`.

## 4. 🔍 Проверка облачных сущностей

Скрипт `scripts/check_entity.py` используется для взаимодействия с API Cloud.Ru без использования Terraform на этапе проверки.

### Параметры CLI:
*   `-a, --api`: Тип API (iam, compute, vpc, nlb и т.д.).
*   `-q, --query`: Сущность (vms, subnets, flavors, vpcs).
*   `-m, --match`: Параметр для фильтрации (например, `name`).
*   `-v, --value`: Значение параметра для фильтрации.

**Пример использования в Ansible:**
```yaml
script: "check_entity.py -a compute -q flavors -m cpu -v 16 -m ram -v 32"
```

## 5. 🎭 Ansible-плейбук и роли

### playbook.yaml
Использует `gather_facts: true` и управляет порядком выполнения через роли. Большинство задач делегируется на `localhost` (управление инфраструктурой).

### Роли:
1.  **`check_inventory`**: Проверяет наличие всех необходимых переменных в CSV и окружении.
2.  **`check_cloudru_entities`**:
    *   Проверяет существование ресурсов в облаке.
    *   Ищет подходящий `flavor` по параметрам CPU/RAM.
    *   **Генерирует Terraform-конфигурации** (`.tf` файлы) из шаблонов в папку `tf/`.
3.  **`terraform`**: Вызывает модуль `community.general.terraform` для применения сгенерированных конфигураций. Ожидает доступности ВМ по SSH.
4.  **`os_config`**:
    *   Настройка репозиториев и установка базовых пакетов.
    *   Создание разделов LVM на `hdd_data_device` и монтирование в `/pub`.
    *   Настройка Chrony и `/etc/hosts`.
5.  **`ipadomain`**:
    *   Ввод в домен FreeIPA.
    *   Генерация JKS кейсторов и трастсторов для Kafka.
6.  **`corax`**:
    *   Распаковка дистрибутива Corax.
    *   Генерация внутренних инвентарей Corax.
    *   Запуск вложенных плейбуков Corax (`prepare_corax.yaml`, `kafka-zookeeper-SE.yml`).
7.  **`corax_post`**: Тонкая настройка после установки (JMX экспортер, Systemd юниты).

## 6. 🌍 Интеграция с Terraform

Интеграция реализована через генерацию кода "на лету":
1.  Роль `check_cloudru_entities` использует Jinja2 шаблоны (`ANSIBLE/roles/check_cloudru_entities/templates/`).
2.  Создаются файлы для каждого ресурса: `subnet_{ogid}.tf`, `{hostname}.tf`, `{hostname}_system_disk.tf` и т.д.
3.  Переменные `CLOUDRU_KEY_ID` и `CLOUDRU_SECRET` передаются в Terraform через префикс `TF_VAR_`.
4.  **Идемпотентность**: Обеспечивается самим Terraform. Если ресурсы уже созданы и параметры не изменились, никаких действий не будет предпринято.

## 7. 🚀 Пошаговое руководство по запуску

### 1. Подготовка
Разместите параметры кластера в папке `INV/` согласно структуре.

### 2. Экспорт переменных
```bash
export TF_VAR_CLOUDRU_KEY_ID="ваш_key_id"
export TF_VAR_CLOUDRU_SECRET="ваш_secret"
export CLOUDRU_PROJECT_ID="ваш_project_id"
export IPA_ADDER="admin_user"
export IPA_ADDER_PASSWORD="password"
# И остальные переменные URL из роли check_inventory
```

### 3. Генерация инвентаря
```bash
python3 ANSIBLE/ansible_prepare_021.py > inventory.yaml
```

### 4. Запуск деплоя
Полный цикл:
```bash
ansible-playbook -i inventory.yaml ANSIBLE/playbook.yaml
```

Запуск отдельных этапов через теги:
*   `--tags tf`: Только работа с инфраструктурой.
*   `--tags os_config`: Только настройка ОС.
*   `--tags corax`: Только установка Corax.

## 9. 🐛 Troubleshooting и логи

### Частые проблемы:
*   **401/403 API Error**: Проверьте корректность `TF_VAR_CLOUDRU_KEY_ID` и `TF_VAR_CLOUDRU_SECRET`.
*   **Flavor not found**: Скрипт `check_entity.py` не нашел flavor с указанными CPU/RAM/Oversubscription. Проверьте доступные квоты.
*   **SSH Timeout**: Проверьте доступность через ProxyJump. Убедитесь, что группа безопасности (Security Group) разрешает входящий SSH порт 22.
*   **IPA Join Failure**: Проверьте доступность DNS серверов и корректность логина/пароля `IPA_ADDER`.

### Отладка:
*   **Python**: Скрипты выводят JSON или ошибки в `stderr`.
*   **Terraform**: Установите `export TF_LOG=DEBUG` перед запуском.
*   **Ansible**: Используйте `-vvv` для детального вывода задач. Логи Terraform сохраняются в `ANSIBLE/tf/terraform.tfstate`.

> ⚠️ **Важно**: Всегда проверяйте содержимое папки `ANSIBLE/tf/` перед запуском Terraform, если вы вручную меняли шаблоны.
