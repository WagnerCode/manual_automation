# Документация по автоматизации развёртывания кластера Corax в Cloud.Ru

Данное руководство описывает систему автоматизированного развёртывания инфраструктуры и кластера Corax (на базе Kafka и Zookeeper) в облачной среде Cloud.Ru.

---

## 1. 📘 Обзор проекта

### Назначение
Автоматизация предназначена для быстрого и воспроизводимого развёртывания кластеров Corax. Она охватывает весь жизненный цикл: от создания виртуальных ресурсов в облаке до финальной настройки прикладного ПО.

### Архитектура решения
Решение использует модульный подход с разделением ответственности между компонентами:
*   **Python (`ansible_prepare_021.py`)**: Предварительная обработка данных. Трансформирует иерархическую структуру каталогов и CSV-файлов в плоский инвентарь Ansible, рассчитывает сетевые адреса и маршруты.
*   **Terraform**: Декларативное управление инфраструктурой. Конфигурации `.tf` генерируются динамически ролью Ansible на основе текущего состояния инвентаря.
*   **Ansible**: Основной оркестратор. Выполняет валидацию, управляет Terraform, настраивает ОС, интегрирует узлы с FreeIPA и устанавливает Corax.
*   **Python (`check_entity.py`)**: Вспомогательная утилита для прямого взаимодействия с API Cloud.Ru (валидация квот, поиск ID ресурсов, проверка существования объектов).

### Data Flow
1.  **Вход**: Статические файлы в каталоге `INV/` (CSV, текстовые файлы `ip`, `access.txt`).
2.  **Трансформация**: Скрипт подготовки генерирует YAML-инвентарь, обогащая его вычисленными данными (Gateway, Prefix, Route files).
3.  **Генерация**: Ansible роль `check_cloudru_entities` считывает инвентарь и заполняет Jinja2 шаблоны Terraform.
4.  **Развёртывание**: Terraform создает VM, диски и сети в Cloud.Ru.
5.  **Конфигурация**: Ansible настраивает ОС и устанавливает Corax, используя данные из инвентаря.

## 2. 🛠 Предварительные требования

### Программное обеспечение
*   **Python 3.x** (библиотеки: `requests`, `pyyaml`, `ipaddress`).
*   **Ansible 2.10+** (коллекции: `community.general`, `ansible.posix`).
*   **Terraform 1.0+** (провайдер `cloudru/cloud` v2.0.0).
*   **Java 17 (OpenJDK)** (требуется для работы с `keytool` при генерации сертификатов).
*   **Системные утилиты**: `sshpass`, `jq`, `openssl`.

### Переменные окружения (Environment Variables)
Система требует установки ряда переменных для аутентификации в Cloud.Ru и FreeIPA:

| Переменная | Описание | Обязательность |
| :--- | :--- | :--- |
| `TF_VAR_CLOUDRU_KEY_ID` | Идентификатор ключа API Cloud.Ru | Да |
| `TF_VAR_CLOUDRU_SECRET` | Секрет ключа API Cloud.Ru | Да |
| `CLOUDRU_PROJECT_ID` | ID проекта в облаке | Да |
| `IPA_ADDER` | Логин для ввода в домен IPA | Да |
| `IPA_ADDER_PASSWORD` | Пароль для ввода в домен IPA | Да |
| `CLOUDRU_*_API_URL` | URL-адреса сервисов (Compute, VPC, IAM и др.) | Да |

> ⚠️ **Важно**: Все переменные `CLOUDRU_*_API_URL` должны быть заданы, так как роль `check_inventory` проверяет их наличие.

## 3. 📁 Структура инвентаря и подготовка данных

Инвентарь строится на основе файловой структуры в каталоге `INV/`.

### Описание файлов
*   **`vm_params.csv`**: Глобальные настройки ВМ (образ, AZ, ресурсы CPU/RAM, параметры дисков).
*   **`net_params.csv`**: Сетевые CIDR для кластера и GIS-подсетей.
*   **`ipa_params.csv`**: Параметры домена FreeIPA.
*   **`dk_ssh_key.csv`**: Публичные ключи для доступа.
*   **Каталоги хостов** (например, `INV/corax_aaaah/host-01/`):
    *   `ip`: Файл с IP адресом управления (ansible_host).
    *   `access.txt`: Строки: 1-user, 2-password, 3-sudo_password, 4-ssh_port.
    *   `eth0.csv`: Специфичные настройки интерфейса (ipaddr, netmask, gateway, route_target).

### Логика `ansible_prepare_021.py`
1.  **Gateway Calculation**: Если в поле `gateway` указано число `n`, скрипт вычисляет n-й IP адрес в подсети. `1` — первый доступный адрес.
2.  **Route Generation**: На основе `route_target` скрипт сопоставляет интерфейсы с целевыми сетями и генерирует содержимое для `route-ethX` файлов, которые позже будут применены в ОС.
3.  **Inventory Mapping**: Все CSV параметры попадают в переменную `config` для каждого хоста в Ansible.

## 4. 🔍 Проверка облачных сущностей (`check_entity.py`)

Скрипт обеспечивает интеграцию с API Cloud.Ru для предварительных проверок.

### Ключевые возможности:
*   **Пагинация**: Автоматически обрабатывает постраничный вывод API (offset/limit), рекурсивно собирая все элементы, пока не будет достигнуто поле `total`.
*   **Матчинг (Matching)**: Позволяет фильтровать результаты по нескольким параметрам одновременно (логическое И).
*   **Endpoints**: Поддерживает широкий спектр API (compute, vpc, iam, kafka, redis, s3 и др.).

### Примеры вызова:
```bash
# Поиск flavor по характеристикам
./check_entity.py -a compute -q flavors -m cpu -v 16 -m ram -v 32 -m oversubscription -v 1:3

# Проверка существования подсети по её адресу
./check_entity.py -a compute -q subnets -m subnet_address -v 10.193.32.0/28
```

## 5. 🎭 Ansible-плейбук и роли

### Порядок выполнения (`playbook.yaml`)
1.  **`check_inventory`**: Валидация всех CSV и env-переменных. Прерывает выполнение при нехватке данных.
2.  **`check_cloudru_entities`**: Поиск ID ресурсов в облаке и генерация `.tf` файлов.
3.  **`terraform`**: Применение (apply) инфраструктуры. Ожидание доступности SSH порта.
4.  **`os_config`**: Базовая настройка ОС (Alt Linux):
    *   Sudoers (беспарольный wheel).
    *   Синхронизация времени (Chrony).
    *   Настройка LVM: `parted` -> `pvcreate` -> `vgcreate` -> `lvcreate` -> `mkfs.ext4` -> `mount` в `/pub`.
    *   Установка системных пакетов (`freeipa-client`, `jq`, `java-17` и др.).
5.  **`ipadomain`**:
    *   Ввод в домен: `ipa-client-install`.
    *   Выпуск сертификата: `ipa-getcert request`.
    *   Создание JKS-хранилищ: сборка PKCS12 из ключей и импорт в JKS через `keytool`.
6.  **`corax`**: Установка продукта. Распаковка архива, подготовка внутреннего инвентаря Corax и запуск его собственных плейбуков через `ansible-shell`.
7.  **`corax_post`**: Установка JMX-экспортера для мониторинга и настройка systemd-юнитов.

## 6. 🌍 Интеграция с Terraform

Ansible выступает в роли "генератора кода" для Terraform.

*   **Шаблоны**: Находятся в `roles/check_cloudru_entities/templates/*.tmpl`.
*   **Ресурсы**: Для каждого хоста создается индивидуальный файл (например, `host-01.tf`), а также общие файлы для сети (`subnet_aaaah.tf`) и групп безопасности.
*   **Управление состоянием**: Terraform state хранится локально в папке `ANSIBLE/tf/`.
*   **Идемпотентность**: Если ВМ уже создана, роль `check_cloudru_entities` проверит её наличие через `check_entity.py`, а Terraform подтвердит соответствие конфигурации.

## 7. 🚀 Пошаговое руководство по запуску

### Шаг 1: Подготовка данных
Заполните CSV файлы в `INV/`. Убедитесь, что файл `ip` содержит актуальный адрес для управления.

### Шаг 2: Генерация инвентаря
```bash
python3 ANSIBLE/ansible_prepare_021.py > inventory.yaml
```

### Шаг 3: Проверка и деплой
Рекомендуется запускать деплой поэтапно с использованием тегов:

1.  **Только инфраструктура**:
    ```bash
    ansible-playbook -i inventory.yaml ANSIBLE/playbook.yaml --tags tf
    ```
2.  **Настройка ОС и домена**:
    ```bash
    ansible-playbook -i inventory.yaml ANSIBLE/playbook.yaml --tags os_config,ipadomain
    ```
3.  **Установка Corax**:
    ```bash
    ansible-playbook -i inventory.yaml ANSIBLE/playbook.yaml --tags corax
    ```

## 9. 🐛 Troubleshooting и логи

### Типовые ошибки
1.  **"Flavor not found"**: Параметры CPU/RAM/Oversubscription в `vm_params.csv` не соответствуют ни одному доступному тарифному плану в Cloud.Ru.
2.  **"Wait for machines online timeout"**: ВМ создана, но недоступна по SSH. Проверьте:
    *   Настройки Security Group (порт 22).
    *   Корректность ProxyJump в инвентаре.
3.  **"LVM creation failed"**: Диск `hdd_data_device` уже имеет разметку или занят. Требуется ручная очистка `wipefs`.

### Где искать логи
*   **Terraform**: `ANSIBLE/tf/terraform.tfstate` и вывод в консоли.
*   **Ansible**: Подробный вывод через `-vvv`.
*   **Corax**: Логи установки находятся на целевых хостах в `/pub/corax/logs/`.
*   **JKS**: Если пароли не совпадают, проверьте файл `ANSIBLE/secrets/ks_secret.txt`.
