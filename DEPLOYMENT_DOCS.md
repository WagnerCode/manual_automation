# Глубокое техническое руководство по автоматизации Corax в Cloud.Ru

Данный документ содержит детальное описание внутренних механизмов, алгоритмов и структур данных, используемых в системе автоматизированного развёртывания кластера Corax.

---

## 1. 📂 Логика инвентаря и генерация данных

### 1.1. Скрипт подготовки `ansible_prepare_021.py`
Скрипт является "точкой входа" для формирования динамического инвентаря Ansible. Он выполняет рекурсивный обход файловой системы в каталоге `INV/` и преобразует плоскую структуру файлов в иерархический YAML.

#### Алгоритм обнаружения хостов:
*   **Хост**: Любой каталог, содержащий файл `ip`, интерпретируется как конечный узел (host). Содержимое файла `ip` становится значением `ansible_host`.
*   **Переменные доступа**: Файл `access.txt` в каталоге хоста (или родительском каталоге) считывается построчно:
    1. `ansible_user`
    2. `ansible_password`
    3. `ansible_become_pass`
    4. `ansible_port`
*   **CSV-параметры**: Все `.csv` файлы в каталоге считываются. Имя файла (без расширения) становится ключом в словаре `config`. Например, `vm_params.csv` порождает `config.vm_params`.

### 1.2. Сетевые вычисления
Скрипт реализует автоматический расчёт сетевых параметров через модуль `ipaddress`.

*   **Расчёт шлюза (`calc_gateway`)**:
    *   Если указано положительное число `n` (например, `1`): шлюз = `network_address + n`.
    *   Если указано отрицательное число `n` (например, `-1`): шлюз = `broadcast_address + n` (последний доступный IP в подсети).
    *   Если `n = 0`: возвращается адрес сети.
*   **Префикс подсети**: Автоматически вычисляется на основе маски через `calc_prefix_len`.
*   **Статическая маршрутизация**: Если в CSV интерфейса указаны `route_target` и `route_gateway`, скрипт формирует словарь `route_files`. Это позволяет автоматически генерировать конфигурационные файлы маршрутов для Alt Linux (формат `ADDRESSn`, `NETMASKn`, `GATEWAYn`).

### 1.3. Формирование `/etc/hosts`
Скрипт собирает данные из всех CSV, где параметр `hosts` установлен в `yes`, и формирует глобальный список `hostsfile` с полными доменными именами (FQDN) на основе переменной `domain`.

## 2. 🤖 Обнаружение ресурсов и генерация IaC

### 2.1. Валидатор API `check_entity.py`
Служебный Python-скрипт для взаимодействия с API Cloud.Ru. Используется Ansible-ролью `check_cloudru_entities` для обеспечения идемпотентности и поиска ID ресурсов.

**Ключевые возможности:**
*   **Авторизация**: Автоматическое получение JWT-токена через IAM API с использованием `TF_VAR_CLOUDRU_KEY_ID` и `TF_VAR_CLOUDRU_SECRET`.
*   **Рекурсивная пагинация**: Поддержка обработки больших списков ресурсов (автоматический инкремент `offset` до тех пор, пока `offset + limit < total`).
*   **Мульти-атрибутивный поиск**: Возможность фильтрации ресурсов по нескольким параметрам одновременно (например, поиск flavor по `cpu`, `ram` и `oversubscription`).

### 2.2. Динамическая генерация Terraform
Вместо статичных `.tf` файлов проект использует Jinja2-шаблоны. Роль `check_cloudru_entities` генерирует конфигурацию в каталог `ANSIBLE/tf/` на основе данных инвентаря:

1.  **`main.tf`**: Провайдер `cloudru` (v2.0.0), переменные и эндпоинты API (compute, vpc, iam и др.).
2.  **`subnet_{ogid}.tf`**: Ресурс `cloudru_evolution_compute_subnet`. Параметры `routed_network` и `default` установлены в `true`.
3.  **`{hostname}.tf`**: Ресурс `cloudru_evolution_compute_vm`. Включает `cloud_init_userdata` в формате Base64.
4.  **`{hostname}_data_disk.tf` / `system_disk.tf`**: Ресурсы `cloudru_evolution_compute_disk`. Системный диск создается из образа (`image_id`).
5.  **`interface_{hostname}.tf`**: Ресурс `cloudru_evolution_compute_interface`.
6.  **`security_group_{ogid}.tf`**: Правила сетевого доступа через `cloudru_evolution_compute_security_group_rule`.

**Сетевая безопасность (Security Group):**
Шаблон `security_group.tmpl` определяет правила:
*   **Входящие (Ingress)**:
    *   SSH (22) из административной подсети (10.246.7.7), Gitlab CI (10.246.0.160/27) и Jump-хостов (10.246.8.0/23).
    *   Порт 9093 для взаимодействия с ГИС (подсеть берется из `config.net_params[0].gis_subnet`).
    *   Полный доступ внутри группы (Intra SG) для всех узлов кластера.
    *   Доступ из технических сервисов Cloud.Ru (JAM, Platform, SUP, IAM SE).
*   **Исходящие (Egress)**:
    *   Доступ к GIS подсети.
    *   DHCP (UDP/67) на адрес 255.255.255.255.

## 3. 🐧 Конфигурация ОС Alt Linux и Storage

### 3.1. Сетевая подсистема (etcnet)
В Alt Linux конфигурация сети управляется через `/etc/net/ifaces/`. Шаблон `vm.tmpl` через `cloud-init` создает для каждого интерфейса:
*   `options`: `BOOTPROTO=static`, `ONBOOT=yes`, `NM_CONTROLLED=no`, `CONFIG_IPV4=yes`.
*   `ipv4address`: Содержит строку вида `IP/PREFIX`.
*   `ipv4route`: Содержит статический маршрут (например, `default via GATEWAY`).

### 3.2. Автоматизация LVM
Роль `os_config` подготавливает диск данных:
1.  **Разметка**: Создание GPT раздела (`parted`) на устройстве из `hdd_data_device`.
2.  **PV/VG/LV**:
    *   Physical Volume: на разделе 1 (напр. `/dev/vdb1`).
    *   Volume Group: `vg_data`.
    *   Logical Volume: `lv_data` (100% объема VG).
3.  **Файловая система**: Ext4.
4.  **Монтирование**: Точка монтирования `/pub`.

> ❗ **Важно**: Роль содержит проверку `config.vm_params[0].cluster_type == "kafka"`. Для других типов кластеров выполнение прерывается, так как логика путей монтирования не определена.

## 4. 🔐 Безопасность и PKI Lifecycle

### 4.1. Ввод в домен FreeIPA
Выполняется через `ipa-client-install` с параметрами:
*   `--mkhomedir`: создание домашних директорий при входе.
*   `--enable-dns-updates`: автоматическая регистрация A-записи в DNS IPA.
*   `--force-join`: принудительное переприсоединение, если хост уже был в домене.

### 4.2. SSL/TLS сертификатов для Corax
Процесс полностью автоматизирован без участия внутренних скриптов Corax:
1.  **Выпуск**: `ipa-getcert request` запрашивает сертификат для `host/FQDN`.
2.  **PKCS12**: Создание бандла `.p12` из полученного CRT и Private Key.
3.  **Java Keystore (JKS)**: Конвертация PKCS12 в JKS с использованием `keytool`.
4.  **Пароли**: Для JKS генерируется случайный пароль, который сохраняется в `ANSIBLE/secrets/ks_secret.txt` на управляющей машине и используется всеми узлами кластера.
5.  **Truststore**: Формируется из всех `.crt` файлов в `roles/ipadomain/files/`.

## 5. 🏗 Оркестрация деплоя Corax

### 5.1. Кастомизация Corax
Система "подменяет" часть логики стандартного инвентаря Corax:
*   **`group_vars/all.yaml`**: Генерируется из `templates/all.tmpl`. Устанавливает протокол безопасности `SSL__ZK_mTLS_WITH_AUTH__KAFKA_SSL_WITH_AUTH__AUDIT` и пути к JKS в `/pub/opt/Apache/kafka/ssl/`.
*   **Патчинг ролей**: Копируются модифицированные файлы `unpack.yml`, `zookeeper_SSL.yml`, `kafka_SSL.yml` и `ansible.cfg`. Это предотвращает перезапись SSL-сертификатов, подготовленных ролью `ipadomain`.

### 5.2. Мониторинг (JMX Exporter)
Роль `corax_post` интегрирует мониторинг:
*   Установка пакета `prometheus-jmx-exporter`.
*   Копирование конфигурации `jmx-exporter.yaml`.
*   Использование скриптов-оберток `kafka-server-start-jmx` для активации Java-агента экспортера на порту **5556**.
*   Регистрация и запуск Systemd-юнита `kafka.service`.

## 6. 📊 Справочник инвентаря (CSV Schema)

### 6.1. `vm_params.csv`
| Поле | Описание |
| :--- | :--- |
| `ogid` | Идентификатор группы (префикс ресурсов). |
| `image_name` | Имя образа ОС (напр. `alt_10.2.1`). |
| `cpu` / `ram` | Ресурсы ВМ. |
| `hdd_data` | Объем диска данных (ГБ). |
| `hdd_data_device` | Путь к устройству (напр. `/dev/vdb`). |
| `oversubscription` | Переподписка CPU (напр. `1:3`). |
| `cluster_type` | Тип (только `kafka`). |
| `dstr_path` / `dstr_file` | Локальный путь к архиву дистрибутива. |

### 6.2. `ethX.csv`
| Поле | Описание |
| :--- | :--- |
| `ipaddr` | IPv4 адрес. |
| `netmask` | Маска в формате `255.255.255.0`. |
| `gateway` | Номер шлюза (1 = первый IP сети). |
| `route_target` | Группа маршрутов (напр. `oam`). |
| `route_gateway` | Шлюз для статических маршрутов группы. |

### 6.3. `net_params.csv`
| Поле | Описание |
| :--- | :--- |
| `subnet` | Основная подсеть кластера (CIDR). |
| `gis_subnet` | Подсеть ГИС для доступа к Kafka (9093). |
